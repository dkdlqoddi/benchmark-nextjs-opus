# p16 — Extract repeated patterns from server actions into shared helpers

## What I extracted and why

Reviewed all three action modules (`actions/auth.ts`, `actions/check-ins.ts`,
`actions/habits.ts`). The auth check and ownership check already lived in shared
helpers (`requireUserId`, `assertHabitOwner` in `lib/auth.ts`); the remaining
duplication was in how the actions *combined* them, parsed forms, and shaped
error responses. Extracted three helpers, all behaviour-preserving:

1. **`requireHabitOwner(habitId)` → `lib/auth.ts`** (auth + ownership check).
   `archiveHabit`, `restoreHabit`, and `deleteHabit` each did the identical
   two-liner `const userId = await requireUserId(); await assertHabitOwner(id, userId);`
   with nothing in between. Collapsed into one call that returns the user id.
   - **Deliberately NOT applied to `updateHabit`/`toggleCheckIn`**: those run
     validation *between* the auth and ownership checks, so collapsing them would
     reorder the checks and change which error surfaces (e.g. an invalid update
     to a non-owned habit must still return field errors, not "Habit not found").
     They keep calling `requireUserId` + `assertHabitOwner` separately.

2. **`parseHabitForm(formData)` → `lib/habit-schema.ts`** (zod parsing + error
   response shape). `createHabit` and `updateHabit` contained the *same* ~13-line
   block: read/trim the form, `habitSchema.safeParse`, parse+validate tags, and
   assemble `{ errors, values }`. Extracted into one function returning a
   discriminated union: `{ ok: true, parsed: { data, tagNames } }` or
   `{ ok: false, state: HabitFormState }`. The private `readForm` and
   `toFieldErrors` helpers moved with it (co-located with the schema they serve).
   `parseHabitForm` is exercised through both create and update, and now via unit
   tests.

3. **`attemptCredentialsSignIn(email, password)` → `actions/auth.ts`** (the
   repeated sign-in try/catch error shape). `loginAction` and `signupAction` both
   wrapped `signIn("credentials", { …, redirectTo: "/" })` in a try/catch that
   distinguishes an `AuthError` (bad credentials) from the `NEXT_REDIRECT` thrown
   on success. Extracted into a helper returning `false` on `AuthError` and
   re-throwing everything else, so each caller keeps its own distinct
   post-failure behaviour (login returns a form error; signup redirects to
   `/login`).

**Behaviour preserved exactly:** all exported action signatures are unchanged;
error messages, return shapes (`{ errors, values }` / `{ error, values }`),
validation order, redirects, and the tag-string outer-trim are identical.
`actions/check-ins.ts` was intentionally left unchanged (its check ordering
doesn't match the combined helper). Net result: `actions/habits.ts` dropped ~94
lines of duplication.

## Changed files

- **`lib/auth.ts`** — added `requireHabitOwner(habitId): Promise<string>`.
- **`lib/habit-schema.ts`** — added `parseHabitForm` + `ParsedHabit` type; moved
  the private `readForm` and `toFieldErrors` here; imports `parseTags`/
  `validateTags` from `lib/tags`.
- **`actions/habits.ts`** — `createHabit`/`updateHabit` now use `parseHabitForm`;
  `archiveHabit`/`restoreHabit`/`deleteHabit` now use `requireHabitOwner`;
  removed the moved helpers. (`resolveTagIds` stays — it's a DB op.)
- **`actions/auth.ts`** — `loginAction`/`signupAction` now use
  `attemptCredentialsSignIn`.
- **`tests/unit/habit-form.test.ts`** (new) — 7 tests locking `parseHabitForm`
  behaviour (valid data + cleaned tags; empty description/tags; name/color/
  targetDays errors with echoed values; too-many-tags; combined field+tag errors).
