# p10 — Per-habit target days of the week

## 1) Work summary

Each habit now has **target days of the week**, stored as a 7-bit weekday mask
(`Habit.targetDays`, bit i = weekday i, 0=Sun..6=Sat; 127 = every day).

- **Model/migration.** Added `targetDays Int @default(127)` to `Habit`. The
  migration is a table rebuild whose row-copy omits `targetDays`, so **every
  existing habit gets the `DEFAULT 127` (= every day)** — requirement satisfied
  at the DB level.
- **Target-day helpers.** New `lib/target-days.ts` (`EVERY_DAY`,
  `isTargetWeekday`, `maskToWeekdays`, `weekdaysToMask`, `toggleWeekday`,
  `isValidMask`, `describeMask`) + `weekdayOfKey()` in `lib/date.ts`.
- **Streak calc counts only target days.** `lib/streak.ts` was rewritten to step
  over target days (via `prevTargetKey`/`nextTargetKey`) instead of raw calendar
  days: non-target weekdays are skipped (they don't break a run), check-ins on
  non-target days never extend a streak, and missing a target day breaks it.
  `computeStats`/`currentStreak`/`longestStreak` now take a `mask` argument. With
  `mask = 127` the algorithm reduces exactly to the previous calendar-day logic,
  so migrated (every-day) habits behave identically to before.
- **Dimmed-but-usable UI on non-target days.** The habit form gained a 7-button
  weekday picker (defaults to every day; validates ≥1 day). The `HabitCard`
  check-in button and the `CalendarMonth` cells/column-headers are **dimmed**
  (`opacity`) on non-target days but remain real submit buttons — you can still
  check in. Cards and the detail page show a target-days label (e.g. "Mon, Wed,
  Fri", "Weekdays", "Every day").
- **Verification script.** `scripts/verify-streak.ts` was rewritten with 10
  cases covering every-day (unchanged behaviour), Mon/Wed/Fri, and Tue/Thu
  masks, including today-is-a-non-target-day, today-is-a-target-day, the "today
  still open" grace, non-target check-ins being ignored, and an 11-in-a-row
  target-day streak. All pass.

## 5) Remaining issues / TODOs

- **Local `prisma/dev.db` still not migrated (unchanged from p09).** `dev.db`
  carries pre-p09 data with no `userId`, so the `add_auth_user_scoping` migration
  can't apply over it, and `prisma migrate reset` is blocked by Prisma's AI-agent
  guardrail without explicit user consent (not granted). `dev.db` is gitignored;
  the new migration is committed, and all verification ran on an equivalent
  freshly-migrated + seeded temp database. To initialize the real dev DB, run
  `npx prisma migrate reset --force` **or**
  `rm prisma/dev.db && npx prisma migrate deploy && npx prisma db seed`.

  days) — intentionally out of scope; the task scoped only the streak calc.
