# p12 — Habit search

## 1) Work summary

Added a **search feature** to the home page. A search box (`?q=`) filters the
user's active habits by matching text against a habit's **name, description, or
tag names** (case-insensitive substring). Search composes with the existing tag
filter (`?tag=`) — both narrow the list together.

- Server-side filtering via Prisma: `OR: [ name contains, description contains,
  tags.some.name contains ]`, AND-ed with the tag filter and the `userId` +
  `archivedAt: null` scoping. SQLite `LIKE` makes it case-insensitive for ASCII.
- The search box is a native `<form method="get" action="/">` (no client JS): it
  navigates to `/?q=…`, and carries the active tag through a hidden field so
  searching keeps the current tag filter. A "Clear" link appears when a query is
  active.
- Filter links stay consistent: the tag chips and "All" are built with a
  `homeHref({ q, tag })` helper so switching tags preserves the search query and
  vice-versa.
- The header summary and empty state reflect the active query/tag (e.g. "1 habit
  tagged “Fitness” matching “morning”", or "No habits match “zzz”.").

No schema or data-model changes were needed.

--- UI ---
  /?q=run                    search input present=yes  Clear link present=yes

All cases behaved as expected.

## 4) Key decisions

- **Integrated into the home page** (not a separate route): the home already
  lists habits and filters by tag, so search belongs there and composes with it.
- **Searches name + description + tag names** so the box is useful regardless of
  how the user remembers a habit.
- **Native GET form** keeps the server-first, no-client-JS approach; the URL is
  shareable/bookmarkable and works without JavaScript.
