# p09 — Authentication with Auth.js (NextAuth v5)

## 1) Work summary

Added email + password authentication (Auth.js / NextAuth v5) with per-user data
isolation.

- **Provider & hashing.** A single Credentials provider (`lib/auth.ts`) verifies
  email + password against a new `User` table; passwords are stored only as
  **bcrypt** hashes (`bcryptjs`, `$2` hashes, cost 10). Sessions are stateless
  JWTs (required for Credentials).
- **Signup / login / logout.** Server Actions in `actions/auth.ts`
  (`signupAction`, `loginAction`, `logoutAction`) drive `signIn`/`signOut`; pages
  at `/signup` and `/login` render a shared client `AuthForm`; logout is a
  Server-Action `<form>` in the top nav.
- **Data model.** New `User` model; `Habit` gains a required `userId` FK
  (`onDelete: Cascade`, indexed). `CheckIn` is scoped transitively through its
  habit. Migration `20260702080341_add_auth_user_scoping`.
- **Route protection (middleware).** Next.js 16 renamed the `middleware` file
  convention to **`proxy`** (now Node.js runtime); `proxy.ts` gates every route:
  logged-out users may reach only `/login` and `/signup`; everything else
  redirects to `/login`, and a logged-in user hitting `/login`/`/signup` is sent
  home.
- **User scoping (defense in depth).** Because Server Actions are POST requests
  to their own route and can bypass the proxy (per the Next 16 proxy docs), auth
  and ownership are re-enforced at the data layer: `requireUserId()` and
  `assertHabitOwner()` in `lib/auth.ts`. Every page read is scoped with
  `where: { userId }`, the detail/edit pages use `findFirst({ where: { id, userId } })`
  → `notFound()` (404) for non-owners, and every habit/check-in mutation calls
  `requireUserId()` + `assertHabitOwner()` before writing.
- **Seed.** `prisma/seed.ts` now creates two accounts — `alice@test.com` and
  `bob@test.com`, password `password123` — each with their own habits and ~2
  weeks of check-ins.

## 3) How the access-blocking was verified, and the result

Because applying the migration to the committed dev database requires a
destructive reset (see §5), verification was run against an **equivalent,
freshly-migrated + seeded temporary SQLite database** (`prisma migrate deploy`
into a throwaway file, then `prisma/seed.ts`). The code paths, queries, and
running build are identical to production — only the DB file differs.

**(a) Data-layer checks** (the exact queries the pages/actions run):

```
PASS  seed: alice has 2 habits, bob has 2 habits
PASS  detail/edit page: alice opening bob's habit id (cmr38a2pl0011pm7aphszh89b) => null => 404
PASS  detail/edit page: bob opening own habit => found
PASS  server actions: assertHabitOwner(bobHabit, alice) has no row => throws (mutation blocked)
PASS  list scoping: alice's home list excludes bob's habits
PASS  bcrypt: correct password verifies
PASS  bcrypt: wrong password rejected
PASS  bcrypt: password stored as a $2 bcrypt hash, not plaintext
ALL ACCESS CHECKS PASSED
```

**(b) Full HTTP end-to-end** against the built app (`next start`) — real proxy,
real NextAuth credentials login, real session cookie:

```
== UNAUTHENTICATED (proxy redirects to /login) ==
  GET /              -> 302 Location: http://localhost:3005/login
  GET /habits/{bob}  -> 302 Location: http://localhost:3005/login
  GET /login         -> 200
  GET /signup        -> 200
== LOG IN AS ALICE (POST /api/auth/callback/credentials) ==
  callback/credentials -> 302 ; session cookie: SET
== AS ALICE (authenticated) ==
  GET /                -> 200
  GET alice's habit    -> 200
  GET bob's habit      -> 404   ← completion criterion met
  GET bob's edit       -> 404
```

**Result:** while logged in as alice, directly requesting bob's habit detail URL
(and edit URL) returns **404**. Unauthenticated access to any protected route
redirects to `/login`; `/login` and `/signup` remain reachable logged-out.

  **one** of:
  - `npx prisma migrate reset --force` (drops, re-migrates, reseeds), or
  - `rm prisma/dev.db && npx prisma migrate deploy && npx prisma db seed`.
- `AUTH_SECRET` is committed in `.env` as a dev-only value (matching the repo's
  existing convention of committing `.env` for reproducibility); rotate it for
  any real deployment.
- The `app/api/auth/[...nextauth]` route is NextAuth's own auth endpoint
  (CSRF/session/callback), not an application data API — app mutations still go
  exclusively through Server Actions per project Rule 2.

  deprecation warning.
