# p14 — Investigate "11:55 PM check-in recorded under the next day's date"

## User report

> "I checked in at 11:55 PM, but the next morning it was recorded under the next
> day's date."

## 1. Investigation process

Traced every code path that assigns a date to a check-in and every path that
displays one:

- **Write path.** `HabitCard` and the calendar submit a Server Action:
  `toggleToday(habitId)` → `toggleCheckIn(habitId, getTodayKey())`
  (`actions/check-ins.ts`). The stored `CheckIn.date` string comes solely from
  `getTodayKey()`. No client value is sent (the action is bound to just the
  habit id / a server-computed key).
- **"Today" computation.** `getTodayKey()` → `toDateKey(new Date())` →
  `seoulParts()` uses `new Intl.DateTimeFormat("en-CA", { timeZone: "Asia/Seoul", … })`
  on the current instant (`lib/date.ts`). The timezone is passed **explicitly**,
  so the result depends only on the absolute instant — it is independent of the
  server's local timezone.
- **Display path.** `app/habits/[id]/page.tsx` and `CalendarMonth` render using
  the stored `date` strings and `todayKey={getTodayKey()}`; `CalendarMonth` is a
  Server Component that compares/derives dates as **plain strings**
  (`weekdayOfKey` splits on `-`; comparisons are lexicographic). There is **no
  client-side `new Date("YYYY-MM-DD")` parsing** anywhere (verified by grepping
  `app/`, `components/`, `lib/`, `actions/`), so there is no local-timezone
  off-by-one in rendering.

Codebase sweep for date/time handling confirmed the only clock reads are
`getTodayKey()`/`currentMonth()` (both Asia/Seoul), and the only other
`new Date()` uses are `Date.UTC(...)`-based helpers (timezone-independent) and
`archivedAt: new Date()` (a timestamp, not a date key).

**Empirical reproduction** — mapped candidate instants through the real
`toDateKey` (`npx tsx`):

```
23:55 KST (Seoul viewer)     2026-07-02T14:55:00Z  ->  2026-07-02   (same day)
23:55 UTC (west of Seoul)    2026-07-02T23:55:00Z  ->  2026-07-03   (next day) ← report
23:55 EDT / UTC-4            2026-07-03T03:55:00Z  ->  2026-07-03   (next day)
Seoul 23:59:59               2026-07-02T14:59:59Z  ->  2026-07-02
Seoul 00:00:00 (midnight)    2026-07-02T15:00:00Z  ->  2026-07-03   (boundary)
```

## 2. Root-cause analysis & conclusion

**Conclusion: this is NOT a code bug.** It is the app's documented, intentional
policy of anchoring all dates to a single fixed timezone (**Asia/Seoul**,
UTC+9), behaving exactly as designed. The code is internally consistent: the
write path and the display path use the *same* Seoul-anchored key, there is no
off-by-one, and there is no dependency on the server's local timezone (`Intl`
receives an explicit `timeZone`).

**Why the report happened.** A check-in is dated to the **Seoul calendar day of
the instant it occurs**. Seoul midnight is **15:00 UTC**. For a viewer **west of
Seoul** (UTC offset < +9 — e.g. UTC, the Americas, most of Europe), late-evening
local time is already past midnight in Seoul:

- 11:55 PM for a **Seoul** viewer = 14:55 UTC → still `2026-07-02` (no rollover).
- 11:55 PM for a **UTC** viewer = 23:55 UTC → `2026-07-03` in Seoul → stored
  under the *next* calendar date, which is what the reporter observed the next
  morning.

So the observation is real and deterministic, but it reflects a **timezone-model
/ UX limitation** (the app has no per-user timezone; `ARCHITECTURE.md` decision
#4 anchors dates to Asia/Seoul), **not a defect** in the date computation. No
functional fix was warranted; changing the anchor to the viewer's local day
would be an architecture change (capture client timezone, thread it into the
Server Action) that contradicts the documented single-timezone design and was
out of scope for this investigation.

## 3. Fixes and tests made

No behavioral code change (the behavior is correct by design). Two changes:

- **`lib/date.ts`** — expanded the `getTodayKey()` JSDoc to state explicitly that
  dates are anchored to Asia/Seoul and that, for a viewer west of Seoul, a
  late-night check-in can be dated to "tomorrow" **by design** — so this is not
  re-triaged as a bug later. Points to the regression tests.
- **`tests/unit/date.test.ts`** — added a `regression: midnight-boundary
  check-in date (p14)` block (3 tests) exercising the real check-in code path
  (`getTodayKey()` with faked system time):
  1. 11:55 PM for a Seoul viewer (14:55 UTC) stays on the same day.
  2. 11:55 PM west of Seoul (23:55 UTC) is dated to the next Seoul day
     (reproduces the report as intended behaviour).
  3. The day switches exactly at Seoul midnight (14:59:59Z → same; 15:00:00Z →
     next), to the tick.
