## 4) Key decisions and reasons

- **Shared form via `useActionState`.** One `HabitForm` client component serves
  create and edit. The action returns `{ errors, values }` on failure (rendered
  inline per field, input preserved) and calls `redirect("/")` on success.
- **Single source of truth for presets/validation.** `lib/colors.ts` holds the
  8 presets; `lib/habit-schema.ts` reuses them in the zod `color` refinement, so
  the picker and server validation can never drift.
- **Minimal `'use client'`.** Archive and restore are plain Server-Component
  `<form action={boundAction}>` (work without JS). Only delete is a client
  component, because it needs a confirmation dialog before mutating.
- **Explicit delete transaction.** `deleteHabit` deletes check-ins then the
  habit in a `$transaction`, making "delete associated check-ins" explicit even
  though the schema's `onDelete: Cascade` would also handle it (both verified).
- **No schema change / migration.** `archivedAt` and the cascade already exist.
- **Verification method.** Create/edit/archive/restore were exercised end-to-end
  over HTTP through the progressive-enhancement Server Action path; delete
  (client-invoked behind a confirm dialog, so not a plain form POST) was verified
  via its exact DB transaction plus the schema cascade, with the rendered Delete
  button confirming the wiring.
