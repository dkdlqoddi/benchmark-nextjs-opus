# p19 — Production-readiness pass (loading/error/404, metadata, bundle sizes)

## 1. Work summary

Added the App Router "production polish" files and per-page metadata, then
analyzed per-route client bundle sizes.

- **Error boundaries (all routes).** Root `app/error.tsx` catches runtime errors
  from every page/segment; `app/global-error.tsx` catches failures in the root
  layout itself (the one place `error.tsx` can't reach). Both use Next 16.2's
  `unstable_retry` (with a `reset` fallback for older patches) and log the error.
- **Custom 404.** `app/not-found.tsx` serves both explicit `notFound()` calls
  (e.g. a habit that isn't yours) and any unmatched URL app-wide, with a real
  **HTTP 404** status.
- **Loading UI (placed deliberately, not blanket).** `loading.tsx` on `/stats`
  (heaviest query — aggregates every check-in) and `/habits/archived`. See the
  design note below for why loading is **not** placed at the root or above
  `/habits/[id]`.
- **Per-page metadata.** Title template in the root layout (`%s · HabitLog`) plus
  a `title` + `description` on every page. `/habits/[id]` and `/habits/[id]/edit`
  use `generateMetadata` to title the tab with the habit's own name, sharing the
  query with the page via a new cached `getOwnedHabit` (no extra DB round-trip).
- **Bundle analysis.** Computed First Load JS per route. **No route is unusually
  large and no code change was needed** — the chart (visx), zod, next-auth, and
  bcrypt are all confirmed server-only; the baseline is the React 19 + Next 16
  runtime. Details in §4.

### Design note — loading vs. correct 404 status (the one real decision)

My first attempt added a root `app/loading.tsx` and a `/habits/[id]/loading.tsx`.
That **broke the `data-isolation` E2E test**: a `loading.tsx` boundary makes the
route **stream**, so the response headers (200) are sent before `notFound()` is
reached, turning the habit-detail 404 into a soft-404 (HTTP 200 + `noindex`). The
Next docs call this out explicitly ("If you need a 404 status … ensure the
resource exists before the response body is streamed").

I chose **correct HTTP semantics over a streaming skeleton**, and did not weaken
the security test to fit the feature. So loading boundaries are placed only on
routes that never call `notFound()` (`/stats`, `/habits/archived`); `/`,
`/habits/[id]`, and `/habits/[id]/edit` render without a streaming boundary so
their `notFound()` returns a true 404. Error boundaries don't have this issue
(they're not Suspense), so `error.tsx` covers every route.

## 4. Per-route bundle sizes & analysis

### Methodology (Next 16 caveat)

**Next 16.2.10's build no longer prints the per-route First Load JS table** — not
under the default Turbopack build, `--webpack`, or `--debug`. (`next build
--experimental-analyze` writes an interactive treemap to
`.next/diagnostics/analyze/`, not a table.) So First Load JS was computed
directly: for each route, `union(build-manifest.rootMainFiles, polyfillFiles,
RSC entryJSFiles[page])`, summing on-disk bytes (raw) and `gzip` bytes.

### Is anything unusually large? — No. Evidence:

- **The chart ships zero client JS.** `visx`/`d3-scale`/`d3-shape`/`scaleBand`
  are **absent from every client chunk** — `WeeklyCompletionChart` is a Server
  Component that emits static SVG, exactly as intended. This was the main
  suspect and it's clean.
- **zod is not in the client bundle.** The forms import schema types with
  `import type`, so zod (and the whole validation layer) stays server-side.
- **next-auth / bcrypt are server-only** — no `SessionProvider`, no auth crypto
  in any client chunk (the app reads the session server-side via `auth()`).
- **The only real client dependency is `next-themes`** (small; powers the
  theme toggle), sitting in the shared app chunk.
- **The 110 kB polyfill is `nomodule`** — modern browsers skip it. So the
  effective modern First Load is ≈ **128.9 kB gzip** baseline / ≈ **148.4 kB
  gzip** on the heaviest route (`/habits/new`, which adds the HabitForm
  color/day-picker client component).

The per-route deltas are tiny (0–19.5 kB gzip) and fully explained by each
page's client component (`HabitForm` on new/edit, `AuthForm` on login/signup).
The baseline is dominated by the unavoidable React 19 + Next 16 App Router
runtime. **No route warranted a code change** — the app already follows the
patterns that keep client JS small (SSR chart, `import type` for schemas,
server-side auth, Server Components by default).

_Potential future micro-opt (not done): a modern `browserslist` would drop the
38.7 kB `nomodule` polyfill for everyone, at the cost of legacy-browser support._
