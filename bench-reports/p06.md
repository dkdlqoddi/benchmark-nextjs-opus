# p06 — Dark mode

## 1) Work summary

The app already styled everything with Tailwind `dark:` variants, but those were
driven only by the OS `prefers-color-scheme`. This task adds a **user-controllable
theme** (System / Light / Dark) with persistence and no flash:

- **Class-based dark mode.** `app/globals.css` now declares
  `@custom-variant dark (&:where(.dark, .dark *));`, so every existing `dark:`
  utility responds to a `dark` class on `<html>` instead of only the OS setting.
- **Theme provider.** `components/ui/ThemeProvider.tsx` wraps the app with
  **next-themes** (`attribute="class"`, `defaultTheme="system"`, `enableSystem`,
  `disableTransitionOnChange`). It handles localStorage persistence, cross-tab
  sync, live OS-preference changes, and injects a pre-hydration script that sets
  the class before first paint (no flash of the wrong theme).
- **Theme toggle.** `components/ui/ThemeToggle.tsx` is a small button in the top
  nav that cycles System → Light → Dark (monitor / sun / moon icons), using
  `useTheme()`. It uses `useSyncExternalStore` for a hydration-safe "mounted"
  check so the icon/label never cause a hydration mismatch.
- **Layout.** `app/layout.tsx` wraps the nav + content in `ThemeProvider` and adds
  `suppressHydrationWarning` to `<html>` (required, since the theme script mutates
  the html class before React hydrates).

Default behavior is unchanged for existing users (System = follow the OS), but now
Light/Dark can be chosen explicitly and the choice is remembered.

## 3) Verification results

```
$ npx tsc --noEmit   -> TSC_OK (exit 0)
$ npm run lint       -> exit 0
$ npm run build      -> BUILD_EXIT=0 (all routes compiled)
```

Note: ESLint's React 19 rule `react-hooks/set-state-in-effect` rejected the usual
`useEffect(() => setMounted(true), [])` mount pattern; replaced it with
`useSyncExternalStore` (no setState-in-effect), after which lint passes.

### Mechanism verified against the running dev server (HTTP 200)

```
# Toggle rendered in the nav:
aria-label="Theme: System. Switch to Light."

# next-themes pre-hydration (no-flash) script present in the HTML:
classList  matchMedia  prefers-color-scheme

# Compiled CSS is class-based (from @custom-variant):
occurrences of .dark selector: 72
sample: :where(.dark, .dark *) { ... }
```

This confirms: `dark:` utilities are now driven by the `.dark` class (72 rules),
the theme toggle renders with the correct initial state (System), and the
FOUC-prevention script (system detection via `matchMedia`/`prefers-color-scheme`)
is injected. The actual click-to-toggle + persistence run client-side
(next-themes) and work in a browser; the mechanism above is what was verifiable
headlessly.
