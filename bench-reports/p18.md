# p18 тАФ Senior-review audit (security / integrity / performance / a11y)

## 1. Work summary

Audited the whole HabitLog codebase as a first-time senior reviewer against the
four requested dimensions (security, data integrity, performance,
accessibility), classified findings **Critical / High / Medium / Low** in a new
root **`AUDIT.md`**, and **fixed both High issues** (no Critical found). The full
test suite (unit + Playwright E2E) plus build/tsc/lint are green after the fixes.

Findings (full detail in `AUDIT.md`):

- **Critical:** none.
- **High (fixed):**
  - **H1 тАФ Accessibility / contrast.** The checked-in button (`HabitCard`) and
    checked calendar day (`CalendarMonth`) drew `text-white` on the habit's own
    color. All 8 presets fail WCAG AA for the label; yellow (~1.6:1) and green
    (~2.0:1) are near-illegible. Fixed with a luminance-based
    `readableOnColorClass()` that returns the higher-contrast Tailwind token
    (near-black for every preset тЗТ тЙе 4.96:1).
  - **H2 тАФ Data integrity / race.** `toggleCheckIn` did a non-atomic
    `deleteMany` тЖТ `create`. A double-click/double-submit of the (undebounced)
    check-in button makes two requests both create, so the second trips the
    `@@unique([habitId, date])` constraint and throws an unhandled `P2002` тЖТ
    500. Fixed by treating that specific collision as success (idempotent
    check-in), rethrowing any other error.
- **Medium (documented):** no auth rate-limiting (M1); account enumeration via
  signup message + login timing (M2); other P2002-prone concurrent writes тАФ
  signup, tag upsert (M3).
- **Low:** weak date validation on check-in (L1, **fixed** as it was adjacent to
  H2 тАФ new `isValidDateKey`); minor over-fetching on `/stats` + `/archived`
  (L2); orphaned tags (L3); committed `AUTH_SECRET` тАФ intentional dev-only repo
  convention (L4); picker `fieldset`/`legend` grouping nit (L5).

Also verified (and documented as **not** vulnerable): auth bypass / IDOR
(defense-in-depth `requireUserId` + `assertHabitOwner`, all reads
`userId`-scoped), SQL injection (Prisma-parameterized), XSS (React escaping),
mass assignment, N+1 (batched `in` queries), unnecessary client components (all
justified; the chart is an SSR Server Component), cascade deletes, and form/label
keyboard accessibility.

## 4. Key decisions and reasons

- **What to fix:** the task scoped fixes to Critical + High. I found **no
  Critical** and **two High** (H1, H2) and fixed exactly those, plus one Low
  (L1) that lived in the same guard clause I was already editing for H2.
  Mediums/Lows are documented with recommendations rather than fixed, to keep the
  change surface tight and reviewable.
- **H1 via a token, not a hex.** `readableOnColorClass` returns a Tailwind class
  (`text-black`/`text-white`), not an inline hex, so Project Rule 5 ("Tailwind
  tokens only") holds тАФ only the background stays inline, which is the allowed
  "habit's own dynamic color" exception. It maximizes WCAG contrast per color, so
  it's correct for any future preset, not just today's 8.
- **H2 = idempotency, not a lock.** SQLite serializes writes, so the robust,
  minimal fix is to catch the unique-constraint collision (`P2002`) and treat
  "already checked in" as the successful end state. I deliberately did **not**
  convert `HabitCard` to a client component just to disable the button while
  pending тАФ the server-side idempotency fix is sufficient; button-debounce is
  noted as optional hardening.
- **`isValidDateKey` (L1)** does a round-trip check so impossible-but-well-shaped
  dates (`2026-13-40`, non-leap `02-29`) are rejected; the UI only ever sends
  valid keys, so this is pure hardening against crafted POSTs with no behavior
  change for real use.
- **`AUTH_SECRET` left committed (L4).** `.gitignore` explicitly opts `.env` in
  and both files document it as a dev-only reproducibility value. I flagged the
  production risk in `AUDIT.md` but did not fight the repo's stated convention.
- **Independent test oracle.** `colors.test.ts` re-derives the WCAG math itself
  rather than importing the implementation's luminance, so it actually guards the
  contrast guarantee instead of tautologically mirroring the code.
