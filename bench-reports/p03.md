# p03 — Daily check-ins + monthly calendar (Asia/Seoul)

## 1) Work summary

- **Check in today** — every home habit card now has a toggle button. It fills
  with the habit's color and reads "✓ Checked in today" when done, and toggles
  the check-in off when pressed again.
- **Monthly calendar** — the new `/habits/[id]` detail page shows a month grid.
  Checked days are filled with the habit's color, today is ringed, and there is
  **Prev / Next month** navigation (via `?month=YYYY-MM`).
- **Toggle from the calendar** — clicking a past or today cell toggles that
  date's check-in; **future dates are disabled** (rendered as non-interactive
  cells, and rejected server-side as a safeguard).
- **Timezone fixed to Asia/Seoul** — all "today"/date logic is centralized in
  `lib/date.ts` (KST via `Intl.DateTimeFormat`), used by the actions, the home
  page, the calendar, and the seed script. Dates remain `YYYY-MM-DD` strings.

Mutations go through Server Actions (`actions/check-ins.ts`). The card toggle and
every calendar cell are plain Server-Component `<form>`s (work without client
JS); no new `'use client'` components were added. No schema change/migration was
needed — `CheckIn` with its `@@unique([habitId, date])` already exists.

Completion criteria met: all four behaviors verified end-to-end over HTTP, and
`npm run build`, `npm run lint`, `npx tsc --noEmit` pass with 0 errors.

## 4) Key decisions and reasons

- **Centralized timezone in `lib/date.ts`.** A single `seoulParts()` (via
  `Intl.DateTimeFormat` with `timeZone: "Asia/Seoul"`) backs `getTodayKey`/
  `toDateKey`; calendar layout math (`weekdayOf`, `daysInMonth`, `buildMonthGrid`)
  uses `Date.UTC` so a calendar date's weekday is timezone-independent. Date keys
  are `YYYY-MM-DD` strings, so past/future comparisons are plain string compares.
- **Server-Component forms for toggles.** The card button and each calendar cell
  are `<form action={boundAction}>`; toggling works without client JS and via
  progressive enhancement, keeping `'use client'` usage at zero for this task.
- **Toggle = delete-or-create.** `toggleCheckIn` runs `deleteMany({habitId,date})`
  and creates only when nothing was deleted, so it flips cleanly and idempotently.
- **Future guard server-side.** Even though future cells aren't rendered as
  buttons, `toggleCheckIn` rejects `dateKey > getTodayKey()` as defense in depth.
- **Month via `?month=YYYY-MM`.** Navigation is plain `<Link>`s, so the calendar
  is shareable/bookmarkable and needs no client state.
- **Checked styling.** Cell background uses the habit's color (dynamic data,
  inline style); the readable text uses the Tailwind `text-white` token — no
  hardcoded colors in code.

## 5) Remaining issues / TODOs

- **Light preset + white text contrast.** Checked cells use white text on the
  habit color; the lighter presets (e.g. yellow) have modest contrast. A
  luminance-based text color could improve accessibility.
- **No-JS UX.** Toggling posts the whole page (fine as progressive enhancement);
  an optimistic client update could feel snappier but isn't required here.
- **`package.json#prisma` deprecation warning** persists (Prisma 6); harmless.
