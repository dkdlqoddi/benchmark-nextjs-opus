# p22 — Critical review: 3 weakest / most regrettable parts

## 1. Work summary

Wrote `REVIEW.md` (kept at the repo root as a separate deliverable) identifying the
three weakest / most regrettable parts of the project, each with a justification
(evidence + impact) and a concrete, phased improvement plan. **No code was
modified** — this was an analysis-only task; the only new file besides this report
is `REVIEW.md`.

Candidates were brainstormed across security, product design, architecture,
performance, and testing, then narrowed to the three with the highest combination of
severity and "regret." Each pick was verified against the current source before
writing (file/line references in `REVIEW.md`):

1. **"Today" is hardcoded to `Asia/Seoul` for every user** (`lib/date.ts:8`,
   `TIME_ZONE`). The day boundary rolls at 15:00 UTC worldwide, so anyone west of
   Korea gets evening check-ins dated to *tomorrow* and streaks that silently break —
   the core "did I do it today?" function is wrong for most users. Most regrettable
   because p14 hit a real user report and responded by adding regression tests that
   pin the wrong behavior as "by design." Plan: add `User.timeZone`, parameterize
   `lib/date`, thread the user's zone through actions/pages, rewrite the p14 tests to
   assert user-local behavior, and handle historical `CheckIn.date` reinterpretation.

2. **`AUTH_SECRET` is committed to a public repo** (`.env` is git-tracked via `!.env`
   in `.gitignore`; `origin` is a public GitHub URL). The session-signing key in
   history means anyone can forge a JWT for any `userId` → total auth bypass, which
   defeats the app's headline per-user isolation. Regrettable because it's a
   deliberate "commit `.env` for reproducibility" convention. Plan: rotate the secret
   (it's burned), stop tracking `.env`, ship a `.env.example`, add a boot-time guard
   against a missing/known secret, and purge history if treated as a real product.

3. **The Server Action / authorization layer has no direct tests.** Verified that no
   test imports anything from `actions/`; the 5 unit files are all pure-lib and the
   E2E suite is exactly 2 tests. The app's most important property (ownership /
   per-user isolation) lives in `actions/*` + `lib/auth.ts` and is the least tested
   code, while the safe streak/date math is exhaustively covered — an investment
   mismatch where a refactor could silently drop an `assertHabitOwner` and stay green.
   Plan: add a Vitest integration project against a throwaway `test.db` (reuse the E2E
   harness), test the full owner/non-owner/unauthenticated matrix for every mutation,
   and lock the p18 fixes (H2/L1, cascade, per-user tags) in as regressions.

`REVIEW.md` also lists four runners-up that were considered and rejected (global
`force-dynamic` + `revalidatePath`; SQLite under multi-user auth; DB invariants
enforced only in the action layer; the Settings placeholder), with a one-line reason
each — to show the top 3 were chosen against alternatives.
